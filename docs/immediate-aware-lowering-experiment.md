# Immediate arithmetic experiments

## Scope

This work was split into two layers so machine encoding capability could be
kept independent from instruction-selection policy.

### E1: machine immediate forms

E1 generalized the machine encoder to support:

- `add/or/and/sub/xor/cmp r64, imm8/imm32`
- `shl/shr/sar r64, imm8`
- `imul r64, r64, imm8/imm32`

The 64-bit ALU and multiply forms accept only values representable by the
architectural sign-extended imm32 encoding. Larger constants remain register
operands. Exact-byte tests cover imm8, imm32, negative values, and extended
register REX bits.

E1 changed encoder capability only. MIR lowering continued to emit its prior
register forms, so existing generated program instruction streams were not
intentionally changed.

### E2: immediate-aware lowering

E2 recognizes a right-hand `MirIntOperand` during MIR-to-X64 lowering and uses
an immediate instruction when it is directly encodable:

- `add/sub/and/or/xor x, C`
- `mul x, C`, using three-operand `imul`
- `shl/sar/shr x, C`
- `icmp.* x, C`

It also uses the same capability for constant GEP offsets and element scales.
Non-encodable constants keep the old `mov rcx, C; op rax, rcx` path.

The final experiment deliberately does not canonicalize constants found on the
left. Right-hand constants dominate the compiler workload, while left-side
canonicalization added enough compiler code to weaken self-host performance.
It also does not perform power-of-two strength reduction; `udiv`, `urem`, and
`mul` rewriting remain a separate experiment.

## Related audits

Compound assignment was inspected while evaluating constant RHS forms, but it
was not changed by E2. The existing dedicated `AstAssignOp` lowering remains
responsible for all compound operators and target kinds.

## Measurements

All bootstrap rows below use the final self-host generation and the same v0
seed. Times are medians of three runs unless noted otherwise.

| Metric | Before E1 | E1 | E2 compact |
|---|---:|---:|---:|
| Total compile time | 3469 ms | 3422 ms | 3453 ms |
| GC stop-the-world | 875 ms | 862 ms | 890 ms |
| X64 items | 180729 | 181824 | 180913 |
| `.text` bytes | 798166 | 802952 | 797634 |
| Final executable | 837120 B | 841728 B | 836608 B |

E1 added encoder implementation code, increasing the self-hosted compiler by
4608 bytes, but showed no runtime regression.

Relative to E1, final E2 produced:

- 911 fewer X64 items
- 5318 fewer `.text` bytes
- a 5120-byte smaller executable
- a 31 ms higher three-run median total time (0.9%), within observed run noise

A five-run alternating own-source comparison reduced batch-order effects:

| Variant | Total | Lower X64 | Machine object | GC STW | Executable |
|---|---:|---:|---:|---:|---:|
| E1 | 3453 ms | 765 ms | 594 ms | 859 ms | 841728 B |
| E2 | 3469 ms | 766 ms | 609 ms | 905 ms | 836608 B |

The 16 ms total difference is 0.46%. GC and machine-stage samples varied by
roughly this scale or more between runs, so the data does not establish a
performance regression. A fixed-source comparison also showed that the code
generated by immediate-aware lowering was not slower while producing a compiler
about 10.5 KiB smaller.

## Conclusion

E1 is a stable machine-layer capability and belongs on `dev`.

E2 has a deterministic code-size benefit and no confirmed wall-time regression.
The compact right-hand-only implementation is preferable to broader versions:
it captures most constant opportunities without making the self-hosted lowering
code disproportionately large. The result is suitable for `dev` and served as
the base for the follow-up
[`power-of-two strength-reduction experiment`](power-of-two-strength-reduction-experiment.md).
That experiment found that the generic optimization cost outweighed its current
self-host benefit, so its code changes were not retained.
